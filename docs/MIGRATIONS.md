# Database Migration Guide

BiblioDrift uses **Flask-Migrate** (backed by Alembic) for all database schema changes. This replaces the previous `db.create_all()` approach with a version-controlled, reversible migration system.

---

## Setup

Install the dependency (already in `config/requirements.txt`):

```bash
pip install Flask-Migrate==4.0.7
```

Set the Flask app environment variable before running any `flask db` command:

```bash
# Windows
set FLASK_APP=backend/app.py

# macOS / Linux
export FLASK_APP=backend/app.py
```

---

## Daily workflow

### Apply all pending migrations (upgrade to latest)

```bash
flask db upgrade
```

Run this every time you pull changes that include new migration files.

### Roll back one migration

```bash
flask db downgrade
```

### Roll back to a specific revision

```bash
flask db downgrade 0001
```

### Roll back everything (drop all tables)

```bash
flask db downgrade base
```

---

## Making a schema change

1. **Edit `backend/models.py`** — add/remove/modify columns, tables, or constraints.

2. **Generate the migration script:**

   ```bash
   flask db migrate -m "add bio column to user"
   ```

   Alembic compares your models against the current database schema and generates a script in `backend/migrations/versions/`.

3. **Review the generated script** in `backend/migrations/versions/`. Always check:
   - `upgrade()` does what you expect
   - `downgrade()` correctly reverses the change
   - No data loss for existing rows (add `nullable=True` or a `server_default` for new non-nullable columns)

4. **Apply the migration:**

   ```bash
   flask db upgrade
   ```

5. **Commit both** `models.py` and the new migration file.

---

## Checking migration status

```bash
# Show current revision applied to the database
flask db current

# Show full migration history
flask db history

# Show pending migrations (not yet applied)
flask db show
```

---

## Migration file naming

Files are stored in `backend/migrations/versions/` and named:

```
<revision_id>_<slug>.py
```

Example: `0002_add_bio_to_user.py`

The revision ID is auto-generated by Alembic. The slug comes from the `-m` message you provide.

---

## Existing database (first-time setup)

If you already have a database created by the old `db.create_all()` approach, stamp it as being at the initial migration without re-running it:

```bash
flask db stamp 0001
```

Then apply any newer migrations normally:

```bash
flask db upgrade
```

---

## Production deployment

Always run migrations as part of your deployment pipeline **before** starting the new app version:

```bash
flask db upgrade
gunicorn backend.app:app
```

This ensures the schema is updated before any new code that depends on it starts serving requests.

## Troubleshooting

**`Target database is not up to date`**
Run `flask db upgrade` to apply pending migrations.

**`Can't locate revision identified by 'xxxx'`**
The migration file is missing. Pull the latest code and try again.

**`Table already exists`**
Your database was created with `db.create_all()` before migrations were introduced. Run `flask db stamp 0001` to mark it as migrated, then `flask db upgrade` for any newer migrations.

**Empty migration generated**
Alembic detected no differences between your models and the database. Check that your model changes are saved and the correct `DATABASE_URL` is set.
